技术建设与理解深度的矛盾

最近和一个朋友深夜聊了很久，聊完之后有种后背发凉的感觉。把一些想法记下来。

起点：一切为了让生活更轻松

半年前我开始搭一套自动化系统，初衷很简单：让工作更轻松，查资料更方便，把重复的事情交给机器。这听起来完全合理，甚至值得鼓励——谁不想省出时间来做更有价值的事呢？

但半年跑下来，我发现一个奇怪的事实：我大部分时间都花在了「建设工具本身」上，而不是真正去理解那些工具要处理的内容。

比如我想研究一个技术方案。正常路径是：读文档、看源码、动手试、搞懂原理。但我的实际路径变成了：先搭一个知识库来管理文档，再写一个 Agent 来自动总结，然后优化搜索流程，再调试 Agent 的提示词……等整个工具链跑顺了，我其实并没有真正读懂那篇文档。

成就感的悄悄转移

更让我不安的是，我的成就感来源发生了变化。

以前，让我兴奋的时刻是「哦，原来是这样！」——某个一直困惑的概念突然通了，某个 bug 找到了根因，某种设计模式突然理解了它存在的理由。那种理解的快感是真实的、深层的。

现在，让我兴奋的时刻变成了「看着 AI 把任务跑完了」。部署流水线绿了，Agent 自动生成了代码，知识库又多了一百条记录。这些都是交付的快感，不是理解的快感。

跟朋友说的时候我原话是：「好久没有过'哦原来是这样'的真正理解时刻了。」说完自己愣了一下。

「不需要重复造轮子」——一句正确的话，和一个危险的滑梯

技术圈有一些深入人心的观念：

  • 「不要重复造轮子」
  • 「站在巨人的肩膀上」
  • 「工具链自动化，人专注在创造性工作上」

这些话本身都对。但它们叠加在一起，产生了一个我没预料到的效果：「理解」这件事，被从重要性序列里悄悄挤了出去。

逻辑链条是这样的：既然有现成的库可以用 → 那我不需要理解它的实现 → 既然 Agent 可以帮我总结 → 那我不需要自己读完 → 既然工具链能自动处理 → 那我不需要知道原理。每一步都合理，每一步都在节省时间，但终点是：我对越来越多的东西，停留在「会用」而不是「理解」的层面。

最可怕的是，这种挤出如此自然，以至于你根本不觉得哪里不对。

最深的恐惧

聊到后面，我和朋友触及了一个更大的焦虑：

如果这个趋势继续下去——不只是我个人，而是整个技术文化的方向——人类会不会逐渐失去理解复杂事物的能力？

不是因为我们变笨了，而是因为环境不再需要我们理解。当所有东西都被封装好、自动化、Agent 化，「理解」就从必需品变成了奢侈品，再变成了「没必要做的事」。一代人教下一代人，传递的不再是原理，而是操作手册。几代人之后，理解的能力就像没有用过的肌肉一样萎缩。

这个词很刺耳，但我们还是说了：人会成为「资产耗材」——消耗资源、产出劳动，但不再真正理解自己在做什么。

自我矛盾

说这些的时候，我意识到一个讽刺的事实：我自己在搭建系统的时候，也在下意识地复制这种文化。

我会想「这个功能不需要告诉用户原理，他们会用就行」，「这个流程封装成黑盒更方便」，「文档不用写太细，Agent 能查」。我在构建的每一层抽象，都在让下一层的人更不需要理解。

我不是在批判工具本身——工具是好的，自动化是好的。我在批判的是一种默认偏好：当「理解」和「效率」冲突时，我们几乎总是毫不犹豫地选择效率，并且觉得这是天经地义的。

所以，为什么要写博客

想了一圈之后，我觉得写博客是我目前能找到的、最朴素也最有效的对抗方式。

不是那种「让 AI 帮我写一篇总结」的博客，而是逼自己慢下来，把做过的事情从头到尾讲清楚的博客。写的过程中你会发现：你以为自己理解的东西，其实说不清楚；你以为忘了的细节，其实在手指放到键盘上的时候，会重新浮现。

写作是一种强制理解。你不能用黑话糊弄过去，不能依赖 Agent 帮你跳过难点，不能封装。一个概念如果你写不明白，就是你还没理解。

我不知道个人的微小抵抗能不能对抗大趋势。但至少，我不想成为那种「什么都会用，什么都不懂」的人。先从写清楚每一篇博客开始吧。
